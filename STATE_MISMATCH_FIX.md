# Fix: OAuth State Mismatch Error

## The Problem

```
❌ State mismatch!
   Saved: Q80onCO5Bx3k0v4HYrnevw
   Received: 7b2a3bb9-3f65-4a98-812f-327a4e86efae
```

**Root Cause**: Client is generating its own `state` parameter instead of using the one from the backend.

---

## How OAuth State Works

The `state` parameter is for **CSRF protection**. It must be:
1. Generated by the **backend** (not client)
2. Saved by the client before redirecting to Google
3. Verified when Google redirects back

## What the Client is Doing Wrong

### ❌ Problem 1: Generating Own State

```javascript
// WRONG - Client generates its own state
const myState = generateRandomString();  // "Q80onCO5Bx..."
localStorage.setItem('oauth_state', myState);

// But backend generates a different state
// Backend: "7b2a3bb9-3f65-4a98-812f-327a4e86efae"
// Client: "Q80onCO5Bx3k0v4HYrnevw"
// These don't match! ❌
```

### ❌ Problem 2: Not Using State from URL

```javascript
// WRONG - Using saved state instead of URL state
const savedState = localStorage.getItem('oauth_state');  // Wrong one!
await fetch(`/callback?code=${code}&state=${savedState}`);
```

## ✅ Correct Implementation

### Step 1: Call Backend to Start Login

```javascript
async function startGoogleLogin() {
  const redirectUri = 'http://localhost:3000/auth/callback';
  const codeVerifier = generateRandomString(128);
  const codeChallenge = await generateCodeChallenge(codeVerifier);

  // Save code_verifier (we'll need it later)
  localStorage.setItem('code_verifier', codeVerifier);

  // Call backend
  const response = await fetch(
    `/auth/client/google/login?redirect_uri=${redirectUri}&code_challenge=${codeChallenge}`
  );
  const data = await response.json();

  // ✅ SAVE THE STATE FROM BACKEND (not your own!)
  localStorage.setItem('oauth_state', data.state);

  // Redirect to Google
  window.location.href = data.auth_url;
}
```

### Step 2: Handle Google Callback

```javascript
async function handleGoogleCallback() {
  // Extract parameters from URL
  const urlParams = new URLSearchParams(window.location.search);
  const code = urlParams.get('code');
  const stateFromUrl = urlParams.get('state');  // ✅ From URL

  // Get saved state
  const savedState = localStorage.getItem('oauth_state');

  // ✅ VERIFY STATE MATCHES
  if (stateFromUrl !== savedState) {
    throw new Error('State mismatch - possible CSRF attack');
  }

  // Get code verifier
  const codeVerifier = localStorage.getItem('code_verifier');

  // ✅ SEND STATE FROM URL (not from localStorage)
  const response = await fetch(
    `/auth/client/google/callback?` +
    `code=${code}&` +
    `state=${stateFromUrl}&` +  // Use URL state!
    `code_verifier=${codeVerifier}`
  );

  const tokens = await response.json();
  // Save tokens...
}
```

## Key Points

1. **Backend generates state** - Client NEVER generates state
2. **Client saves backend's state** - Before redirecting to Google
3. **Google returns same state** - In the callback URL
4. **Client verifies state matches** - Checks URL state vs saved state
5. **Client sends URL state to backend** - Not the saved one

## Common Mistakes

### Mistake 1: Using Your Own State Generator

```javascript
// ❌ DON'T DO THIS
import { generateState } from 'some-oauth-library';
const myState = generateState();
```

The backend already generates the state. Don't generate your own!

### Mistake 2: Not Extracting State from URL

```javascript
// ❌ DON'T DO THIS
const savedState = localStorage.getItem('oauth_state');
await fetch(`/callback?state=${savedState}`);  // Wrong!
```

You must use the `state` from the URL, not from localStorage!

### Mistake 3: Clearing State Too Early

```javascript
// ❌ DON'T DO THIS
localStorage.removeItem('oauth_state');  // Before calling /callback
await fetch('/callback?...');
```

Keep the state until after you verify it in the callback!

## Summary

**Issue**: Client generating own state instead of using backend's

**Fix**:
1. Save `state` from backend response
2. Extract `state` from URL after Google redirects
3. Verify they match
4. Send URL `state` to backend callback

**The state is generated by BACKEND, not CLIENT!**
